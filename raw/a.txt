オープンソースコミュニティと法人格の関係を調べると、「法人格は何のためにあるのか？」という問いに行き着きます。

結論から言うと、法人格の役割は主に以下の5つです。 ([osr.finos.org][1])

1. 商標を保有する（名前を守る）
2. 著作権や知財を保有する
3. 寄付や会費を受け取る
4. 法的責任を引き受ける
5. ガバナンスを制度化する

ただし、どこまで法人が介入するかはかなり違います。

---

# 類型1: コミュニティを支える「傘」としての財団

代表例:

* Apache Software Foundation
* Software Freedom Conservancy

このタイプは、

> コミュニティが主体
> 財団は法務・会計・商標管理を担当

という構図です。

Apacheでは個別プロジェクトにPMC(Project Management Committee)があり、技術判断は各プロジェクトが行います。財団は法的・組織的な基盤を提供します。 ([Opensource.com][2])

これはかなり「自治体モデル」に近い。

---

# 類型2: 中立的な業界コンソーシアム

代表例:

* [Linux Foundation](https://www.linuxfoundation.org?utm_source=chatgpt.com)

Linux Foundationは誤解されやすいのですが、

「Linuxのコミュニティ財団」

というより

「企業連合が共同でOSS基盤を維持するための中立機関」

です。 ([Linux Foundation][3])

特徴:

* 会費を払う企業が多数参加
* Linuxだけでなく多数のプロジェクトを管理
* Kubernetes
* PyTorch
* OpenTofu
* Hyperledger
* Overture Maps

などを抱えている。 ([Linux Foundation][3])

近年は

> 特定企業から独立させるためにLinux Foundationへ移管

というパターンが増えています。Redis→ValkeyやTerraform→OpenTofuなどが有名です。中立財団に移すことで、企業支配への不信を下げる狙いがあります。 ([arXiv][4])

---

# 類型3: 財団＋営利企業の二層構造

代表例:

* [Mozilla Foundation](https://www.mozillafoundation.org?utm_source=chatgpt.com)

Mozillaはかなり興味深い。

構造としては

```
Mozilla Foundation (非営利)

    ↓所有

Mozilla Corporation (営利)
```

です。 ([Mozilla][5])

財団が理念を保持し、

* Firefox開発
* マーケティング
* 契約
* 収益事業

はMozilla Corporationが行う。 ([ウィキペディア][6])

これは

> 理念を非営利法人に固定しながら
> ビジネスは株式会社で行う

という設計です。

---

# 類型4: プロトコルの守護者

代表例:

* Ethereum Foundation

Ethereum FoundationはLinux FoundationともMozillaとも少し違う。

Ethereumそのものを支配しているわけではありません。

むしろ

> プロトコルの長期発展を支援するための資金供給機関

です。

* 研究助成
* クライアント開発支援
* イベント開催
* 標準化支援

が主業務。

Ethereumの実装は

* Geth
* Nethermind
* Besu

など複数組織が担当しています。

そのため

> Foundationは重要だが支配者ではない

という位置づけです。

これは学術コミュニティの財団に近い。

---

# 類型5: BDFLを支える法人

代表例:

* Linus Torvalds と Linux
* Guido van Rossum と Python（歴史的には）

ここでは実際の権威は個人にあり、

法人は周辺支援を行います。

Linux Foundationが存在しても、

Linuxカーネルの技術的最終判断は長らくLinusが行っています。 ([ウィキペディア][7])

# 最近の重要な変化

2010年代までは

```
企業
 ↓
OSS公開
 ↓
コミュニティ形成
```

だった。

しかし近年は

```
企業所有
 ↓
ライセンス変更
 ↓
コミュニティ反発
 ↓
フォーク
 ↓
中立財団へ移管
```

が増えています。

代表例:

* Elasticsearch → OpenSearch
* Terraform → OpenTofu
* Redis → Valkey

です。 ([arXiv][4])

この流れの中でLinux Foundationは

> 中立ガバナンスの受け皿

としての役割が強くなっています。 ([SoftwareSeni][8])

---

## 1. 結論

DD2030における法人格は、単なる「OSS開発団体」では足りません。

必要なのは、次のような役割を持つ法人です。

> **政治的中立性・データ責任・契約責任・資金管理・OSS公共財の継続性を担う、信託的な器**

ここで重要なのは、法人がコミュニティを所有することではありません。
むしろ逆で、法人はコミュニティ・自治体・政党・市民・開発者・研究者のあいだに立つ、**法務・会計・手続き・信用のインターフェース**になるべきです。

DD2030は、公式に「特定団体が運営を担うのではなく、一人ひとりが対等に関わる、緩やかな参加型コミュニティ」を掲げています。また、政治的中立性についても、特定政党・候補者・政治団体への支持不支持を行わない「政党中立」と、誰に対しても同一条件・同一手続きで関わる「手続き的中立性」を区別して説明しています。([デジタル民主主義2030][1])

この構想を現実に運用するには、法人が必要です。ただしその法人は「中心政府」ではなく、**公共財の管理者**であるべきです。

---

## 2. DD2030の現在地

DD2030は、すでに単なる構想段階ではありません。

公式サイト上では、主なプロジェクトとして、広聴AI、いどばた、Polimoney、Project Coreloop が示されています。広聴AIは多数の市民の声を収集・可視化する取り組み、いどばたは大規模熟議プラットフォーム、Polimoneyは政治資金の透明化、Project Coreloopは市民の声から政策・実行・フィードバックまでの循環を扱うものとして説明されています。([デジタル民主主義2030][2])

また、新規参加者向けページでは、2025年3月にSlackが開始され、リポジトリ公開とOSSコミュニティ形成が進み、2026年時点では「構想の立ち上げ」を越えて実験・実装フェーズに入っていることが示されています。([デジタル民主主義2030][3])

さらに、自治体・政党・企業・団体などがPoCに参加できる導線、寄付の受付、GitHubとSlackを通じた開発参加の導線も用意されています。([デジタル民主主義2030][4])

つまりDD2030は、すでに次の4つを同時に扱っています。

| 領域     | 内容                    |
| ------ | --------------------- |
| OSS    | 広聴AI、いどばた等の開発         |
| 公共利用   | 自治体・政党・団体でのPoC        |
| 政治的中立性 | 政党・候補者・政治団体との距離の設計    |
| 市民データ  | 意見、政治的関心、対話ログ、分析結果の扱い |

この段階では、法人格がない、または法人の役割が曖昧なままだと、外部から見た信用、責任範囲、データ保護、資金の透明性が不安定になります。

特に気になるのは、現時点でDD2030公式サイト上のプライバシーポリシーと利用規約が「準備中」と表示されている点です。市民の意見や政治的関心に関わるデータを扱い、自治体・政党・団体での実証を進めるなら、これは早期に整備すべき基礎インフラです。([デジタル民主主義2030][5])

## 3. 参考になる先行事例

DD2030にそのまま当てはまる単一の先行例はありません。
近いのは、**Decidim + OpenStreetMap Foundation + Apache Software Foundation + Ethereum Foundation** の混合型です。

Linux Foundation や Mozilla Foundation も参考になりますが、DD2030の性質を考えると、単純にそれらを模倣するのは危険です。

---

### 3.1 Apache Software Foundation：法人とプロジェクト自治の分離

Apache Software Foundationでは、法人としてのASFが存在しつつ、個別プロジェクトの技術的意思決定はProject Management Committee、つまりPMCに委ねられています。理事会は法人全体の方針や法的責任を持ち、PMCがコミッター選任、リリース、技術運営を担う構造です。([Apache Software Foundation][6])

DD2030への示唆は明確です。

法人がすべてを決めるのではなく、たとえば次のように分けるべきです。

| 層         | 役割                        |
| --------- | ------------------------- |
| 法人理事会     | 法務、会計、中立性、リスク、外部契約        |
| プロジェクト委員会 | 広聴AI、いどばた、Polimoney等の技術運営 |
| コミュニティ    | Issue、提案、実装、利用知見、改善要求     |

DD2030にとってApache型から学ぶべき点は、**法人の存在と、開発コミュニティの自治を分離すること**です。

---

### 3.2 Linux Foundation：中立的な受け皿だが、企業連合色が強い

Linux Foundationは、多数のOSSプロジェクトを中立的にホストし、開発者や企業が共同でオープン技術を維持するための場を提供しています。公式には、オープンソフトウェア、ハードウェア、標準、データを中立的にホストする役割が強調されています。([Linux Foundation][7])

DD2030にとって参考になるのは、プロジェクトホスティング、商標管理、参加企業・団体との関係整理、標準化、ポリシー整備です。

一方で、Linux Foundationは企業参加を前提にした業界コンソーシアムとしての性格が強い。DD2030は政治的・市民的な公共圏を扱うため、企業・政党・自治体の利用を受け入れるとしても、**資金提供者や大口利用者が方向性を支配しているように見えない設計**が必要です。

CNCFには、財団のミッションに直接関わる場合を除き、政治的声明や政治的支持を行わないという政治的中立性ポリシーがあります。([CNCF][8])
DD2030の場合は政治そのものを対象にするため、これよりさらに精密な中立性規程が必要です。

---

### 3.3 Mozilla Foundation：理念を守る非営利法人と、事業を担う会社の二層構造

Mozillaは、非営利のMozilla Foundationと、その完全子会社であるMozilla Corporationを持つ二層構造です。Foundationが公共的ミッションとガバナンスを担い、CorporationがFirefoxなどの製品開発・事業運営を担う形です。([Mozilla][9])

DD2030では、将来的に次のような状況が起きる可能性があります。

* 自治体向けにホスティングや運用支援を提供する
* 政党や団体向けに有償サポートを提供する
* 研修、ファシリテーション、データ分析支援を提供する
* セキュリティ・監査・SLAを伴うサービスを提供する

この場合、理念を守る法人と、サービス提供を行う事業体を分けることは検討に値します。

ただし、初期段階から複雑な二層構造にする必要はありません。まずは非営利志向の法人を作り、事業規模が大きくなった段階で、Mozilla型の分離を検討するのが現実的です。

---

### 3.4 Ethereum Foundation：支配者ではなく、エコシステムの支援者

Ethereum Foundationは、Ethereumを所有・支配する存在ではなく、助成、研究、イベント、標準化、エコシステム支援を通じて、Ethereumが公共財として存続する条件を整える存在だと説明されています。Ethereum自体は特定の個人や組織に所有されるものではなく、変更はエコシステム内の合意を通じて進むという考え方です。([ethereum.foundation][10])

DD2030にとって重要なのはここです。

法人が強くなりすぎると、次のような誤解が生じます。

> 「DD2030の法人が、市民の声を代表している」
> 「法人が民主主義の正当性を持っている」
> 「法人がAI分析結果を政策判断として認定している」

これは避けるべきです。

法人の役割は、**民主的正当性を代行することではなく、民主的手続きを支える道具・記録・説明責任を整えること**です。

Ethereum型から学ぶべきなのは、法人を「支配者」ではなく「保全者」「資金供給者」「調整者」として設計することです。

---

### 3.5 OpenStreetMap Foundation：草の根公共財を支える組織

OpenStreetMap Foundationは、OpenStreetMapの草の根性を保ちながら、サーバー、法務、ライセンス、データ、ローカルチャプター、ワーキンググループなどを支える組織です。メンバーが理事を選び、各種ワーキンググループが運営を担う構造になっています。([welcome.openstreetmap.org][11])

DD2030に近い点は、公共財を中央集権的に所有するのではなく、コミュニティの活動を維持する「骨格」を法人が支えていることです。

DD2030でも、将来的には次のような構造が考えられます。

| 機能     | ワーキンググループ例                |
| ------ | ------------------------- |
| 技術     | 広聴AI、いどばた、Polimoney       |
| データ    | プライバシー、匿名化、保存期間、公開範囲      |
| 公共導入   | 自治体・議会・学校・市民団体との連携        |
| 法務     | 契約、規約、商標、CLA              |
| ガバナンス  | 中立性、利益相反、意思決定プロセス         |
| コミュニティ | Slack、GitHub、イベント、モデレーション |

OpenStreetMap型は、DD2030の「緩やかな参加型コミュニティ」という自己理解と相性が良いです。

---

### 3.6 Decidim：最も近い参照例

DD2030に最も近いのはDecidimです。

Decidimは、市民参加のための自由・オープンなデジタルプラットフォームであり、透明性、追跡可能性、完全性、セキュリティ、プライバシー、機密性などを重視するものとして説明されています。([Decidim][12])

さらに、DecidimにはFree Software Associationがあり、Decidimコミュニティのガバナンスを担う民主的な協会として位置づけられています。そこでは、機会の平等、オープンコンテンツ、自由ソフトウェア、データの機密性、透明性、追跡可能性、完全性、継続的改善などが価値として示されています。([Decidim][13])

これはDD2030にとって非常に重要です。

DD2030も、単なるOSSではなく、民主主義の手続きを支えるインフラを作っています。したがって、ライセンスやリポジトリ管理だけでは不十分で、**民主的手続きの保証原則**が必要です。

Decidimから学ぶべきなのは、次の点です。

| Decidimからの示唆 | DD2030で必要な対応        |
| ------------ | ------------------- |
| ソーシャルコントラクト  | DD2030憲章・公共性原則      |
| 自由ソフトウェア     | OSSライセンスと再利用可能性     |
| 透明性・追跡可能性    | 意見収集、AI分析、政策反映過程の記録 |
| プライバシー・機密性   | 個人情報・政治的意見データの保護    |
| コミュニティガバナンス  | 法人、開発者、市民、利用団体の関係整理 |

DD2030が法人格を持つなら、単なる定款だけでなく、Decidimのような**社会契約文書**が必要です。

---

### 3.7 vTaiwan / g0v：法人よりもプロセス設計が重要な例

vTaiwanは、オンラインとオフラインの対話を組み合わせ、市民・政府・専門家・企業・市民社会が公共課題を議論する分散的な公開協議プロセスです。([info.vtaiwan.tw][14])
また、g0vは台湾の分散型シビックテックコミュニティで、情報透明性、成果の公開、オープンな協働を核にしています。([g0v.tw][15])

ここからの示唆は、法人格だけでは民主的正当性は生まれないということです。

必要なのは法人そのものではなく、次のようなプロセスです。

* 誰が参加できるのか
* どの意見がどう集約されるのか
* AIが何をしたのか
* 人間がどこで判断したのか
* 少数意見はどう扱われるのか
* 政策や意思決定にどう接続されたのか
* 最終的に採用されなかった意見にどうフィードバックするのか

法人はこのプロセスを実装・記録・公開する責任を持つべきです。

---

## 4. DD2030に必要な法人機能

DD2030の文脈では、法人格に必要な機能は次の8つです。

---

### 4.1 中立性を制度化する

DD2030はすでに、政党中立性と手続き的中立性を掲げています。([デジタル民主主義2030][1])
しかし、理念として掲げるだけでは不十分です。

法人としては、少なくとも次の文書が必要です。

| 文書        | 内容                       |
| --------- | ------------------------ |
| 中立性ポリシー   | 特定政党・候補者・政治団体を支持しないこと    |
| 利用提供ポリシー  | 自治体・政党・団体に同一条件で提供する基準    |
| 利益相反ポリシー  | 理事・職員・主要貢献者の政治活動・契約関係の開示 |
| 発言ポリシー    | 法人名義で政治的意見を出す範囲          |
| 選挙期間対応ルール | 公職選挙法リスク、報酬、表示、利用停止条件    |

DD2030のコントリビューターガイドラインでは、DD2030が特定政党・候補者・政治団体を支持・反対しないこと、個人の政治活動とDD2030での活動を分けること、DD2030の名称・リソースを個人の政治活動に使わないことが示されています。([デジタル民主主義2030][16])

これは良い出発点です。ただし法人運営では、より具体的に「誰が、いつ、何を開示し、違反時にどう処理するか」まで必要になります。

---

### 4.2 プライバシー・データ保護を中核に置く

DD2030のツールは、市民の意見、政治的関心、地域課題、政策選好に近いデータを扱います。これは一般的なWebサービスの利用ログよりも慎重に扱うべきです。

広聴AIの説明ページ自体も、AI分析に過度に依存すべきではないこと、オンライン意見には偏りがありうること、分析結果をそのまま政策に反映すべきではないことを明記しています。([デジタル民主主義2030][17])

法人として必要なのは、単なるプライバシーポリシーではなく、次のようなデータガバナンスです。

| 項目    | 必要な内容                  |
| ----- | ---------------------- |
| 収集    | 何を収集するか、なぜ必要か          |
| 同意    | 誰が何に同意したのか             |
| 保存    | 保存期間、保存場所、削除方法         |
| 利用    | AI分析、研究利用、公開利用の範囲      |
| 第三者提供 | 自治体・政党・研究者・開発者との共有条件   |
| 匿名化   | 再識別リスクへの対応             |
| 公開    | オープンデータ化する場合の条件        |
| 監査    | 分析結果、プロンプト、モデル、分類基準の検証 |
| 苦情処理  | 削除請求、訂正請求、異議申立て        |

プライバシーポリシーと利用規約が「準備中」のまま見える状態は、実証実験や公共導入の拡大前に解消した方がよいです。([デジタル民主主義2030][5])

---

### 4.3 OSS公共財としての知財・商標・ライセンスを管理する

いどばたのGitHubでは、DD2030が主導・運営するOSSとして、政党・自治体・企業・学校・コミュニティなどが自前でホスト・運用することを想定し、特定政党や特定組織に依存しない公共財を目指すと説明されています。ライセンスはGPL-3.0です。([GitHub][18])

また、いどばたではCLAが用意されており、コントリビューターは著作権を保持しながら、DD2030およびプロジェクトに対して著作権・特許に関する広範なライセンスを付与する形になっています。([GitHub][19])

これは法人格と強く関係します。
CLAの相手方、商標の保有者、ドメイン管理者、リポジトリ管理者、リリース責任者が曖昧だと、長期的な公共財として不安定になります。

法人が担うべきものは次です。

| 対象      | 法人が担う理由                 |
| ------- | ----------------------- |
| 商標      | 「DD2030」「いどばた」等の名前の濫用防止 |
| ドメイン    | なりすまし・政治利用の防止           |
| CLA     | 貢献コードの権利関係整理            |
| ライセンス方針 | OSSとしての再利用可能性維持         |
| リリース    | 公式版と派生版の区別              |
| セキュリティ  | 脆弱性報告・修正・告知             |
| ブランド利用  | 「DD2030公認」等の誤認防止        |

Linux Foundationも、オープンソースライセンスは商標利用許諾とは別であり、商標は品質や出所の混同を防ぐために必要だと説明しています。([Linux Foundation][20])
DD2030でも、コードは自由に使えても、「公式」「認定」「DD2030準拠」という表示は制御する必要があります。

---

### 4.4 契約主体になる

DD2030は、自治体、政党、企業、団体によるPoC参加を受け付けています。([デジタル民主主義2030][4])

この段階では、契約主体が必要です。

必要になる契約・文書は次のようなものです。

| 文書         | 用途                     |
| ---------- | ---------------------- |
| PoC契約書     | 実証実験の目的、期間、費用、成果物、責任範囲 |
| データ処理契約    | 個人情報・意見データの処理責任        |
| 利用規約       | ユーザーとサービス提供者の関係        |
| プライバシーポリシー | データ利用の説明               |
| セキュリティ基準   | アクセス制御、ログ、インシデント対応     |
| SLA        | 有償提供時の稼働保証・サポート範囲      |
| 研究利用規程     | 論文・分析・公開データ利用          |
| 成果公開規程     | 実証結果の公開範囲              |

法人がない場合、これらの責任が個人や任意団体に乗ります。
これは継続性・責任能力・外部信用の観点で弱いです。

---

### 4.5 資金の透明性を担保する

DD2030は、特定の組織や個人に依存せず、広く社会から寄付を集めることで持続可能性を高めたいと説明しています。([デジタル民主主義2030][1])

この方針は、政治的中立性と直結します。

必要なのは、単に寄付を受け取る口座ではありません。
次のような資金ルールです。

| 項目           | 必要な設計                |
| ------------ | -------------------- |
| 寄付受入基準       | 受け取らない寄付の条件          |
| 大口寄付開示       | 金額、団体名、利害関係          |
| 匿名寄付         | 上限・扱い                |
| 企業協賛         | ロゴ掲出、発言権、ロードマップ影響の制限 |
| 政党・政治団体からの資金 | 原則、例外、公開基準           |
| 助成金          | 助成元と成果物の独立性          |
| 支出公開         | 人件費、外注費、クラウド費、イベント費  |
| 監査           | 会計報告、第三者チェック         |

日本での法人格としては、初期には一般社団法人が実務上扱いやすい候補になります。一般社団法人は、設立時社員が2人以上必要で、定款認証などの手続きにより設立されます。([法務省][21])
一方、寄付税制や公益性を前面に出すなら、将来的に公益社団法人・公益財団法人、あるいは認定NPO法人等も検討対象になります。公益法人やNPO法人には寄付税制上の扱いに違いがあります。([内閣府][22])

ただし、これは法務・税務の専門家確認が必要です。設計上の論点は、法人種別そのものよりも、**資金が中立性を損なわないように見えるか**です。

---

### 4.6 民主的手続きの品質保証を担う

DD2030は、「民意を聞く」だけではなく、収集、分析、熟議、政策接続、実行、フィードバックまでの循環を目指しています。Project Coreloopでは、市民の声から合意形成、政策・法令・予算への反映、実行後のフィードバックまでを扱う構想が示されています。([coreloop.dd2030.org][23])

ここで法人が担うべきなのは、結果の正しさの保証ではありません。
担うべきなのは、**手続きの品質保証**です。

具体的には次です。

| 論点        | 問い                     |
| --------- | ---------------------- |
| 参加者代表性    | 誰の声が入っていて、誰の声が入っていないか  |
| バイアス      | オンライン参加者に偏っていないか       |
| AI要約      | どのモデル・プロンプト・分類基準で要約したか |
| 少数意見      | 多数派に吸収されず残っているか        |
| ファシリテーション | 議論が誘導されていないか           |
| 政策接続      | 分析結果がどう意思決定に使われたか      |
| フィードバック   | 採用・不採用の理由が返されているか      |

大阪大学SSIの共同研究プロジェクトでも、広聴AIやDecidim等を組み合わせ、市民の声の収集、AIによる分析、熟議、政策反映、フィードバックまでを一貫したサイクルとして設計する方向が示されています。([Osaka University SSI][24])

法人は、このサイクルを実装する際の「方法の説明責任」を負うべきです。

---

### 4.7 コミュニティの持続性を支える

DD2030のSlackには多数の参加者がいて、技術、実装、思想、理論、書籍執筆などが並行して行われていることが示されています。([デジタル民主主義2030][3])

この規模になると、コミュニティ運営も自然発生だけでは難しくなります。

必要なのは次です。

| 項目              | 内容                       |
| --------------- | ------------------------ |
| Code of Conduct | ハラスメント、攻撃、差別、荒らしへの対応     |
| モデレーション         | Slack、GitHub、イベントでの基準    |
| Maintainer制度    | 権限、選任、退任、後継              |
| ワーキンググループ       | 技術・法務・データ・公共導入など         |
| 参加導線            | 初心者、研究者、自治体職員、開発者それぞれの入口 |
| 意思決定記録          | 重要な設計変更のログ               |
| 紛争処理            | 方針対立、政治的対立、権限濫用への対応      |

OSSはコードだけではなく、維持する人間関係の設計でもあります。

---

### 4.8 外部から見える説明責任を持つ

DD2030が公共インフラに近づくほど、外部から見える説明責任が重要になります。

最低限必要なのは次です。

| 公開物      | 内容            |
| -------- | ------------- |
| 年次報告     | 活動、導入事例、失敗、学び |
| 会計報告     | 収入、支出、寄付、助成   |
| ガバナンス報告  | 理事、委員会、利益相反   |
| セキュリティ報告 | 脆弱性対応、インシデント  |
| データ報告    | 収集・保存・削除・研究利用 |
| 利用団体一覧   | 自治体、政党、団体、企業  |
| 契約方針     | 同一条件提供の基準     |
| ロードマップ   | 何を優先し、なぜ優先するか |

Team Miraiの政策提案ページでも、デジタル民主主義インフラの運営を一政党から切り離し、独立したNGOや超党派的な枠組みで運営すべきという提案や、政治的中立性・OSS化・透明性を求めるコメントが示されています。([policy.team-mir.ai][25])

これはDD2030法人にとって重要な外部要請です。

## 5. 法人が担うべきこと／担うべきでないこと

### 担うべきこと

| 領域   | 法人の役割                    |
| ---- | ------------------------ |
| 法務   | 契約、規約、個人情報、知財、責任範囲       |
| 会計   | 寄付、助成、会費、支出、監査           |
| 中立性  | 政党中立、手続き的中立、利益相反管理       |
| OSS  | ライセンス、CLA、商標、リリース、セキュリティ |
| データ  | プライバシー、匿名化、保存、削除、公開      |
| 公共導入 | 自治体・政党・団体との標準契約          |
| 説明責任 | 年次報告、会計報告、導入実績、失敗事例      |
| 継続性  | Maintainer支援、インフラ費用、人材継承 |

### 担うべきでないこと

| 避けるべきこと               | 理由                  |
| --------------------- | ------------------- |
| 「市民の総意」を名乗る           | 法人に民主的代表権はない        |
| AI分析を政策判断として認定する      | AIは補助であり、正当性の源泉ではない |
| 特定政党・候補者の実質的支援になる     | 中立性を損なう             |
| 大口資金提供者にロードマップを左右させる  | 公共財への信頼を損なう         |
| コミュニティの技術判断をすべて理事会が握る | OSSの自律性を損なう         |
| 公式ブランドを無制限に使わせる       | 誤認・政治利用・品質低下を招く     |
| 実証結果を成功事例としてのみ公表する    | 公共技術としての学習を阻害する     |

法人は、民主主義の主体ではなく、**民主的な試行錯誤を安全に行うための制度的足場**です。

## 6. 推奨する組織設計

現時点での最も自然な設計は、次のようなものです。

```text
デジタル民主主義2030
= コミュニティ、OSSプロジェクト群、実証ネットワーク、思想・研究の場

デジタル民主主義推進機構
= 法務・会計・契約・中立性・知財・データ責任を担う法人
```

そのうえで、内部構造は次のように分けるのがよいです。

```text
法人理事会
  - 法務、会計、リスク、中立性、資金、外部契約

監事・外部監査
  - 会計、利益相反、法人運営のチェック

公共性・倫理委員会
  - 政治的中立性、データ利用、AI利用、公共導入の審査

プロジェクト委員会
  - 広聴AI
  - いどばた
  - Polimoney
  - Project Coreloop

コミュニティ・ワーキンググループ
  - 開発
  - デザイン
  - ファシリテーション
  - 自治体導入
  - 研究
  - ドキュメント
```

特に重要なのは、**理事会が技術判断を独占しないこと**です。
Apache型に近く、プロジェクトごとの技術委員会・Maintainer制度を置くのがよいです。

一方で、中立性、資金、契約、プライバシー、商標、セキュリティについては、法人が明確に責任を持つべきです。

## 7. 初期に優先すべき整備項目

優先順位をつけるなら、次です。

### 第1段階：法人の最低限の信頼基盤

| 優先度 | 項目                      |

| 高   | 中立性ポリシー                 |
| 高   | 利益相反ポリシー                |
| 高   | プライバシーポリシー              |
| 高   | 利用規約                    |
| 高   | 寄付受入ポリシー                |
| 高   | 役員・意思決定構造の公開            |
| 高   | GitHub・ドメイン・商標・CLAの帰属整理 |

この段階では、「美しい理念」よりも、外部の自治体職員・市民・開発者・寄付者が見て、責任主体を理解できることが重要です。

---

### 第2段階：公共導入に耐える運用基盤

| 優先度 | 項目            |
| --- | ------------- |
| 高   | PoC契約テンプレート   |
| 高   | データ処理契約テンプレート |
| 高   | セキュリティポリシー    |
| 高   | インシデント対応手順    |

自治体・政党・市民団体で使われる場合、「誰の責任で」「何を保証し」「何を保証しないか」を明確にする必要があります。

### 第3段階：コミュニティガバナンス

| 優先度 | 項目              |

| 中   | コミュニティからの議題提案制度 |
| 中   | 透明なロードマップ決定手続き  |

この段階で、法人とコミュニティの関係を明文化します。

### 第4段階：公共財としての成熟

| 優先度 | 項目               |
| --- | ---------------- |
| 中   | 年次報告書            |
| 中   | 会計報告             |
| 中   | 外部監査             |
| 中   | 導入事例・失敗事例の公開     |
| 中   | 第三者評価            |
| 中   | 認定制度または準拠表示制度    |
| 低〜中 | 公益法人化・認定NPO化等の検討 |

ここまで来ると、法人は単なる事務局ではなく、公共技術の信頼基盤になります。

## 8. 法人格の選択について

現実的には、初期の器としては**一般社団法人**が有力です。

理由は次です。

* 設立が比較的速い
* 理事会・社員総会などの制度を設計しやすい
* 契約主体になれる
* 寄付・助成・会費・事業収入を扱える
* OSS財団的な運営に向いている

ただし、一般社団法人には「公益性が自動的に保証される」わけではありません。外部からの信頼を得るには、定款、役員構成、会計公開、利益相反管理、寄付ポリシーによって公益性を示す必要があります。

将来的に、広く寄付を集め、税制上の優遇や公益性の明確化が必要になれば、公益社団法人・公益財団法人、または認定NPO法人も検討対象になります。公益法人やNPO法人では寄付税制や認定要件が異なるため、これは事業計画と資金計画に応じて選ぶべきです。([内閣府][22])

私見では、初期段階では次が妥当です。

```text
第1段階:
一般社団法人として設立
  - 非営利型を意識した定款
  - 中立性・利益相反・寄付ポリシーを厚くする
  - OSSと公共実証の契約主体になる

第2段階:
寄付・助成・公共導入が増えた時点で
  - 公益認定
  - 認定NPO
  - 事業子会社
  - 財団型構造
を比較検討する
```

## 9. もっとも注意すべきリスク

### 9.1 「中立性」が空文化するリスク

DD2030は政治を扱うため、中立性は理念ではなく運用問題です。

特に次が危険です。

* 特定政党が先行利用しているように見える
* 特定候補者の宣伝に近い使われ方をする
* 理事・主要メンバーの政治活動が法人活動と混ざる
* 寄付者が政策方向に影響しているように見える
* AI分析結果が「民意」として政治利用される

必要なのは「中立です」と言うことではなく、**同一条件・同一手続き・公開記録**です。

---

### 9.2 AIが正当性を過剰に帯びるリスク

広聴AIやいどばたは、意見を整理し、見えにくい論点を発見するには有用です。
しかし、AIが要約したものを「市民の意思」と見なすと危険です。

DD2030自身も、広聴AIの説明で、AI分析に過度に依存すべきではなく、オンライン意見には偏りがあり、政策に直接反映すべきではないと明記しています。([デジタル民主主義2030][17])

法人は、AIの結果ではなく、次を保証すべきです。

* 入力データの範囲
* 分析方法
* 不確実性
* 少数意見の扱い
* 人間の判断箇所
* 政策反映の過程
* 反映しなかった理由

---

### 9.3 OSSと公共運用の責任範囲が混ざるリスク

OSSとして公開することと、公共サービスとして安全に運用することは別です。

いどばたは、自前ホストや自由な改変を想定するOSSとして説明されています。([GitHub][18])
その場合、法人は次を区別する必要があります。

| 区分     | 責任                     |
| ------ | ---------------------- |
| OSSコード | ライセンス、脆弱性対応、リリース       |
| 公式ホスト版 | 利用規約、プライバシー、可用性、セキュリティ |
| 自治体導入版 | 契約、データ処理、運用責任分担        |
| 派生版    | 商標利用、公式認定の有無           |

コードを公開しているだけなら、責任範囲は比較的限定できます。
しかし公式サービスとして提供するなら、利用者保護とデータ責任が重くなります。

---

## 10. まとめ

DD2030の法人格に必要なのは、Linux Foundationのような大型コンソーシアムをそのまま作ることではありません。Mozillaのように理念法人と事業会社をいきなり分けることでもありません。

現時点で最も近い設計は、次です。

```text
Apache型:
  法人とプロジェクト自治を分ける

OpenStreetMap型:
  草の根公共財を法人が支える

Decidim型:
  民主的手続きの保証原則を明文化する

Ethereum Foundation型:
  支配者ではなく、エコシステムの保全者になる
```

したがって、デジタル民主主義推進機構に必要なことは、端的には次です。

> **DD2030を所有する法人ではなく、DD2030が公共財として存続するための法的・会計的・倫理的・技術的な信頼基盤になること。**

優先順位としては、まず以下を整えるべきです。

1. 定款・ミッション・中立性ポリシー
2. 利益相反ポリシー
3. プライバシーポリシーと利用規約
4. 寄付・資金提供ポリシー
5. OSS知財・商標・CLA・リリース責任の整理
6. 自治体・政党・団体向けPoC契約テンプレート
7. データガバナンスとAI分析の説明責任
8. プロジェクト委員会・Maintainer制度・ワーキンググループ
9. 年次報告・会計報告・導入事例・失敗事例の公開
10. 将来的な公益法人化・認定NPO化・事業子会社化の検討

一言で言えば、DD2030の法人格は、**民主主義を名乗る主体**ではなく、**民主的な実験が社会から信頼されるための制度的足場**として設計するのがよいです。

# NISHIO

いや、どうするべきかという提案を求めてたんではなくて、コミュニティと法人が並走する状況をいまいち理解しにくい人もいるかなと思って解説があるといいかなと思った

# GPT

なるほど、必要なのは「どう設計すべきか」より、まず **“コミュニティと法人が並走する”とはどういう状態なのか** を説明することですね。

この文脈では、次の一文が芯になると思います。

> **デジタル民主主義2030は、人々が参加して育てるコミュニティであり、デジタル民主主義推進機構は、その活動が社会と接続するための法人格である。両者は同じものではないが、互いに支え合う。**

以下、その説明として使える形で整理します。

---

# コミュニティと法人は、そもそも役割が違う

オープンソースやシビックテックでは、しばしば次の二つが同時に存在します。

| 種類     | 何か               | 得意なこと                     |
| ------ | ---------------- | ------------------------- |
| コミュニティ | 人々の参加・議論・開発・実践の場 | アイデア、実装、改善、知見共有、参加の広がり    |
| 法人     | 法律上の責任主体         | 契約、会計、寄付、雇用、知財、規約、対外的説明責任 |

コミュニティは、人の集まりです。
法人は、法律上の人格です。

この二つは重なりますが、同一ではありません。

たとえばDD2030について言えば、公式サイトでは「広聴AI」「いどばた」「Polimoney」などの取り組みが示され、新規参加者向けページでは「3つの柱＋大きなコミュニティ」として説明されています。つまりDD2030は、単一の会社や部署というより、複数のプロジェクトと参加者からなる活動体です。([デジタル民主主義2030][1])

一方で、外部社会との接点では、どうしても法人格が必要になります。寄付を受け取る、契約を結ぶ、ドメインや商標を管理する、利用規約やプライバシーポリシーを置く、自治体や団体と実証実験を行う、といったことは「ゆるやかな人の集まり」だけでは扱いにくいからです。

---

# なぜ「コミュニティだけ」では足りないのか

コミュニティだけでも、コードを書く、議論する、ドキュメントを書く、イベントをする、といった活動はできます。

しかし次のような場面では、外部からこう問われます。

> 誰と契約すればよいのか。
> 誰に寄付すればよいのか。
> 誰が責任を持つのか。
> 誰が商標やドメインを管理しているのか。
> 個人情報や政治的な意見データは誰が管理するのか。

この問いに答えるために法人格が必要になります。

逆に、法人だけでも足りません。
法人があっても、参加者がいなければOSSは育ちません。利用者からのフィードバックも集まりません。多様な現場知も入ってきません。

したがって、公共的なOSSプロジェクトでは、

```text
コミュニティが活動を生む
法人がその活動を社会に接続する
```

という二層構造になりやすい。

ここを誤解すると、「法人が全部を運営しているのか」「コミュニティは法人の下部組織なのか」「法人がDD2030を所有しているのか」という疑問が出ます。実際には、そう単純ではありません。

---

# 「並走」とは、上下関係ではなく役割分担である

並走という言葉で説明したいのは、法人がコミュニティの上に立つわけではなく、コミュニティが法人を単なる事務局として完全に従属させるわけでもない、ということです。

イメージとしては、こうです。

```text
          社会・自治体・団体・寄付者・利用者
                         ↑
             契約・会計・規約・責任
                         ↑
        デジタル民主主義推進機構
              法人格・対外窓口・信用
                         ↑
     ------------------------------------
                         ↓
        デジタル民主主義2030コミュニティ
       開発・議論・実践・改善・知見共有
```

法人は、コミュニティを「所有」するというより、コミュニティの活動を社会に通すための変換器です。

コミュニティ側で生まれた成果や議論を、外部社会が扱える形にする。
外部から来る資金・契約・要請・責任を、コミュニティが壊れない形に整理する。

この両方向の接続が法人の役割です。

---

# オープンソースでは珍しい構造ではない

この構造はDD2030特有ではなく、オープンソースではかなり一般的です。

## Apacheの場合

Apache Software Foundationでは、法人としての理事会が存在しますが、個別プロジェクトの技術的運営はProject Management Committee、つまりPMCに委ねられます。Apacheの説明では、理事会が法人上の方針や委任を行い、PMCがコミッター選任、プロジェクトごとの方針、リリース決定などを担います。([apache.org][2])

ここで重要なのは、法人があるからといって、理事会がすべての技術判断を直接行うわけではない点です。

```text
Apache Software Foundation = 法人格・全体の器
各Apacheプロジェクト = 技術コミュニティ
PMC = 両者をつなぐ責任ある委員会
```

これは「法人とコミュニティの分業」の典型例です。

---

## Linux Foundationの場合

Linux Foundationは、自らを「開発者や組織がオープン技術プロジェクトを開発・管理・拡張するための中立的なハブ」と説明しています。([Linux Foundation][3])

さらにLinux Foundation日本語サイトのプロジェクトホスティング説明では、プロジェクトがOSI承認ライセンスを使うこと、ドメインや商標などのコミュニティ資産をLinux Foundationがプロジェクトコミュニティのために保有できること、技術コミュニティへの参加にメンバーシップは不要であることが説明されています。([The Linux Foundation][4])

ここでも、

```text
法人が資産や対外的信用を持つ
技術コミュニティは開かれている
```

という分離があります。

つまり、商標やドメインを法人が持つことは、コミュニティを法人が所有することとは限りません。むしろ、特定企業や特定個人に握らせないために、法人が中立的な保管者になる場合があります。

---

## Mozillaの場合

Mozillaは、Mozilla Foundationという非営利法人と、Mozilla Corporationという事業会社を持っています。Mozillaの公式説明では、FoundationがMozillaコミュニティを支え、ガバナンス構造を監督し、インターネットを公共資源として守る活動を行うとされています。([Mozilla][5])

この例では、さらに構造が分かれています。

```text
Mozillaコミュニティ
Mozilla Foundation
Mozilla Corporation
Firefoxなどの製品・事業
```

ここでも「Mozilla」という名前が指すものは一つではありません。
コミュニティ、理念、法人、製品、事業体が重なって存在しています。

DD2030でも、似た混乱は起きやすいです。
「DD2030」と言ったとき、それがコミュニティを指すのか、プロジェクト群を指すのか、法人を指すのか、OSSリポジトリを指すのかを分けて説明する必要があります。

---

## Ethereum Foundationの場合

Ethereum Foundationは、Ethereumそのものを所有する会社ではありません。公式の沿革では、初期のEthereumエコシステムが生き残る条件を整えるために、集まりを組織し、初期インフラに資金を出し、研究開発を支援し、実験を育てた存在として説明されています。([Ethereum Foundation][6])

ここでの関係は、

```text
Ethereum = プロトコルとエコシステム
Ethereum Foundation = それを支援する法人
```

です。

法人が重要であることと、法人がすべてを支配していることは別です。

この区別は、DD2030にも近い。
デジタル民主主義推進機構が存在するからといって、それが「市民の声」や「デジタル民主主義そのもの」を代表するわけではない。支援し、守り、社会との接点を担う法人だと説明する方が誤解が少ないです。

---

## OpenStreetMapの場合

OpenStreetMap Foundationも近い例です。OpenStreetMap Foundationでは、ワーキンググループが財団を支える主要な方法として説明されており、財団の内外から参加者を歓迎しています。([osmfoundation.org][7])

ここでは、

```text
OpenStreetMap = 地図データを作る世界的コミュニティ
OpenStreetMap Foundation = サーバー・法務・作業部会などを支える法人
```

という関係です。

地図を作っているのはコミュニティです。
しかし、サーバー、法務、ライセンス、対外的な調整には法人が必要になります。

DD2030でも同じで、民主主義の実践やツールの改善はコミュニティが担う。けれど、公共利用・契約・寄付・責任の受け皿には法人が必要になる。

---

## Decidimの場合

DD2030に特に近いのはDecidimです。Decidimは、市民参加のための自由・オープンソースのデジタル基盤として説明されています。また、Decidim Free Software Associationは、Decidimコミュニティのガバナンスを担う民主的な協会だと説明されています。([decidim.org][8])

これは、単なるソフトウェア開発ではなく、市民参加の手続きそのものを支える基盤です。

DD2030も、いどばたを「民主主義における議論・合意形成を支援するOSS」とし、政党・自治体・企業・学校・地域コミュニティなどが自らホスティングして活用すること、また特定の政党や団体に依存しない公共財として提供することを説明しています。([GitHub][9])

このため、DD2030では「OSSコミュニティ」と「公共的な法人格」の両方が必要になります。
コードだけの話ではなく、公共圏との接続が含まれるからです。

---

# DD2030の文脈で言い換えると

DD2030については、次のように説明すると理解しやすいと思います。

```text
デジタル民主主義2030
= コミュニティ、プロジェクト群、思想、実践、OSS開発の場

デジタル民主主義推進機構
= その活動を社会に接続するための法人格
```

もう少し平たく言うと、

```text
DD2030は「みんなで育てる場」
推進機構は「社会とやりとりするための器」
```

です。

DD2030の公式説明では、政治的中立性について、特定政党・候補者・政治団体の活動に加担しない党派中立と、誰に対しても同一条件・同一手続きで関与する手続き中立が示されています。また、個人メンバーの政治的信条や活動は尊重されるとも説明されています。([デジタル民主主義2030][10])

この点がまさに、コミュニティと法人を分けて説明する必要がある理由です。

個人参加者は、多様な政治的意見を持っていてよい。
しかし、法人としてのDD2030または推進機構は、特定政党や候補者の活動主体になってはいけない。

つまり、

```text
個人の自由
コミュニティの多様性
法人としての中立性
```

を同時に成立させる必要があります。

この三つを混ぜると混乱します。

---

# 混乱が起きるポイント

多くの人が混乱するのは、「運営」という言葉が広すぎるからです。

たとえば「DD2030を運営しているのは誰か？」という問いには、実は複数の答えがあります。

| 問い                  | 答え                            |
| ------------------- | ----------------------------- |
| コードを書いているのは誰か       | 開発コミュニティ、Maintainer、コントリビューター |
| 議論や知見を育てているのは誰か     | 参加者全体                         |
| 外部契約を結ぶのは誰か         | 法人                            |
| 寄付を受け取るのは誰か         | 法人                            |
| 商標やドメインを管理するのは誰か    | 法人または法人が定めた管理主体               |
| プロジェクトの方向性に助言するのは誰か | ボードや有識者、コミュニティ                |
| ツールを現場で使うのは誰か       | 自治体、団体、政党、学校、地域コミュニティなど       |

「運営」という一語で全部をまとめると、法人が全部を決めているようにも見えるし、逆に誰も責任を持っていないようにも見えます。

だから、説明では「運営」ではなく、役割ごとに分けるのがよいです。

---

# 法人は“代表者”ではなく“責任の受け皿”

ここはかなり重要です。

デジタル民主主義推進機構が法人格を持つとしても、それは「DD2030コミュニティ全員の意見を代表する」という意味ではありません。

法人が代表できるのは、法人としての行為です。

* 契約する
* 寄付を受け取る
* 会計報告する
* ドメインを管理する
* 利用規約を定める
* プライバシーポリシーを掲げる
* 公式サービスを提供する
* OSSプロジェクトの公式窓口になる

これらについては法人が責任を持てます。

しかし、次のようなものを法人がそのまま代表するわけではありません。

* 参加者一人ひとりの政治的意見
* コミュニティ全体の思想
* 市民全体の民意
* AIが要約した意見の正当性
* 派生プロジェクトや外部利用者の全行為

この区別を置くと、「法人があるならコミュニティは形だけなのか？」という疑問に答えやすくなります。

法人は、コミュニティの上位者ではない。
ただし、社会と接続する場面では責任主体になる。

---

# 逆に、コミュニティは法人の“部門”ではない

もう一つの誤解は、法人ができると、コミュニティが法人の内部組織のように見えてしまうことです。

しかしOSSでは、コミュニティは法人の従業員だけで構成されるものではありません。Linux Foundationのホスティング説明でも、技術コミュニティへの参加にメンバーシップは不要とされています。([The Linux Foundation][4])

DD2030でも、いどばたはOSSとして、政党・自治体・企業・学校・地域コミュニティなど多様な主体が自らホスティング・運用することを想定していると説明されています。([GitHub][9])

つまり、コミュニティは法人の内側だけに閉じません。

```text
法人の中にコミュニティがある
```

というより、

```text
広いコミュニティの一部として法人がある
```

と見た方が近いです。

法人は重要な役割を担いますが、コミュニティの全体を囲い込むものではありません。

---

# 説明文としては、このくらいが使いやすい

外部向けに説明するなら、次のような文章がよさそうです。

> デジタル民主主義2030は、広聴AI、いどばた、PolimoneyなどのOSS開発や実践を進める参加型コミュニティです。ここでは、開発者、研究者、自治体関係者、市民、デザイナー、ファシリテーターなど、多様な人々が参加し、議論し、改善を重ねています。
>
> 一方で、こうした活動を社会の中で継続していくには、契約、会計、寄付、知財、ドメイン管理、利用規約、プライバシー対応などを担う法律上の主体が必要になります。その役割を担うのが、法人格としてのデジタル民主主義推進機構です。
>
> したがって、デジタル民主主義2030とデジタル民主主義推進機構は同じものではありません。前者は人々が参加して育てるコミュニティであり、後者はその活動が社会と接続するための法人格です。法人はコミュニティを所有するものではなく、コミュニティの活動を継続可能にするための責任の受け皿です。
>
> この関係は、オープンソースの世界では珍しいものではありません。Apache、Linux Foundation、OpenStreetMap、Decidimなどでも、参加者によるコミュニティと、法務・会計・知財・対外窓口を担う法人が役割分担しながら並走しています。

# さらに短く言うなら

もっと短い説明なら、これです。

> DD2030は「活動の場」であり、推進機構は「責任の器」です。
> コミュニティは、議論し、作り、使い、改善します。
> 法人は、契約し、会計を持ち、規約を整え、外部に対して責任を持ちます。
> この二つは上下関係ではなく、公共的なOSSを継続するための役割分担です。

あるいは、さらに一文で言うなら、

> **コミュニティが公共財を育て、法人がその公共財を社会の中で扱える形にする。**

この説明がいちばん誤解を減らしやすいと思います。
